# JSON‑Atom Format Specification

**Version:** 0 (Draft)

**Status:** Draft — subject to change before v1.

---

## 1. Introduction

### 1.1 Purpose

JSON‑Atom Format defines a language-agnostic format for representing atomic changes to JSON documents. A JSON‑Atom Format document (a "delta") describes how to transform a **source** document into a **target** document through an ordered sequence of operations.

### 1.2 Scope

This specification defines:

- The structure of a delta document
- Three operations: `add`, `remove`, `replace`
- A path expression syntax for addressing locations within JSON documents
- Rules for applying operations and reversing them
- Conformance levels for implementations

### 1.3 Non-Scope

This specification does **not** define:

- How to compute a delta from two documents (diff algorithms are implementation concerns)
- How to detect changes (comparison strategies — structural, semantic, or otherwise)
- How to resolve conflicts when merging concurrent deltas
- How to transport or store deltas

### 1.4 Relationship to Existing Standards

**RFC 6902 (JSON Patch)** defines a similar patch format using RFC 6901 JSON Pointer for paths. JSON‑Atom Format differs in two key ways: (1) it uses JSONPath-based expressions that support identity-based array element selection, and (2) it supports optional `oldValue` fields for reversibility.

**RFC 7396 (JSON Merge Patch)** represents changes by overlaying partial documents. It cannot represent atomic array element changes, cannot distinguish `null`-as-value from `null`-as-delete, and does not support reversibility.

**RFC 9535 (JSONPath)** defines a query language for JSON. JSON‑Atom Path (Section 5) is a constrained subset of JSONPath limited to expressions that resolve to exactly one location.

### 1.5 Design Principles

1. **Format over algorithm.** The spec defines what a delta looks like, not how to produce one.
2. **Atomic operations.** Each operation is self-contained and targets exactly one location.
3. **Self-describing paths.** Array identity is embedded in the path — no external schema needed.
4. **Self-identifying documents.** The `format` field makes deltas discoverable without context.
5. **Strict JSON only.** Values are RFC 8259 JSON. No undefined, no Date objects, no functions.
6. **Extension-friendly.** Unknown properties are preserved; `x_`-prefixed properties are future-safe.
7. **Reversibility is derived.** A delta is reversible when all `replace`/`remove` operations include `oldValue`.
8. **Canonical paths.** One correct way to write a path. Accept both canonical and non-canonical; produce canonical.

## 2. Conventions and Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119).

| Term | Definition |
| --- | --- |
| **Document** | A JSON value conforming to [RFC 8259](https://www.rfc-editor.org/rfc/rfc8259). |
| **Source** | The document before changes are applied. |
| **Target** | The document after changes are applied. |
| **Atom** | A JSON‑Atom Format document containing an ordered sequence of operations. |
| **Operation** | A single atomic change at a specific path. |
| **Path** | A JSON‑Atom Path expression (Section 5). |
| **Reversible delta** | A delta where every `replace` and `remove` operation includes `oldValue`. |

## 3. JSON Value Constraints

All `value` and `oldValue` fields in a delta MUST be valid JSON values as defined by [RFC 8259](https://www.rfc-editor.org/rfc/rfc8259).

The supported JSON value types are:

- object
- array
- string
- number
- boolean (`true`, `false`)
- null

The JavaScript value `undefined` is NOT a valid JSON value. Absent properties have no representation in JSON‑Atom Format — they are simply not present.

Implementations MUST NOT produce deltas containing non-JSON values (e.g., `undefined`, `Date`, `RegExp`, `Function`, `Symbol`, `BigInt`). Implementations that process JavaScript objects containing such types SHOULD serialize them to JSON-safe representations before computing deltas, or reject them with an error.

**Example — `null` is a valid JSON value:**

```json
{ "op": "replace", "path": "$.metadata", "value": "active", "oldValue": null }
```

This operation replaces the `null` value at `$.metadata` with the string `"active"`. Both `null` and `"active"` are valid JSON values.

Note: `undefined` is not a JSON value. A delta that would contain `undefined` is malformed — the property should simply be omitted. Absent properties have no representation in JSON‑Atom Format; they are expressed through `add` and `remove` operations.

## 4. Delta Document Format

A delta document MUST be a JSON object containing the following fields:

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `format` | string | REQUIRED | MUST be the literal string `"json-atom"`. Format discriminator for self-identification. |
| `version` | integer | REQUIRED | Format version number. Currently `1`. Incremented only on breaking changes. |
| `operations` | array | REQUIRED | Ordered array of operation objects (Section 6). |

### 4.1 Additional Properties

A delta document MAY contain additional properties beyond `format`, `version`, and `operations`. Conformant implementations MUST preserve these properties and MUST NOT reject a delta because of their presence.

Properties prefixed with `x_` are reserved for extensions. Future versions of this specification will NEVER define a property starting with `x_`. This is the guaranteed-safe namespace for extension authors.

Properties without the `x_` prefix that are not defined by the current specification version are also preserved, but MAY be claimed by future specification versions.

### 4.2 Empty Operations

An empty `operations` array (`[]`) is a valid delta representing a no-op: the source document equals the target document.

### 4.3 Example

```json
{
  "format": "json-atom",
  "version": 1,
  "operations": [
    { "op": "replace", "path": "$.user.name", "value": "Anna", "oldValue": "Alice" },
    { "op": "add", "path": "$.user.role", "value": "admin" },
    { "op": "remove", "path": "$.settings.legacyMode", "oldValue": true }
  ]
}
```

**Invalid delta — missing required `format` field:**

```json
{
  "version": 1,
  "operations": [
    { "op": "replace", "path": "$.user.name", "value": "Anna" }
  ]
}
```

This is not a valid JSON‑Atom Format document. The `format` field is REQUIRED and MUST be `"json-atom"`.

## 5. JSON‑Atom Path

A JSON‑Atom Path is a string expression that identifies exactly one location within a JSON document. It is a constrained subset of JSONPath as defined in [RFC 9535](https://www.rfc-editor.org/rfc/rfc9535).

### 5.1 Grammar

A JSON‑Atom Path is composed of a root symbol followed by zero or more segments:

```
path          = "$" *segment
segment       = dot-property / bracket-property / array-index / key-filter / value-filter

dot-property     = "." property-name
property-name    = (ALPHA / "_") *(ALPHA / DIGIT / "_")

bracket-property = "['" escaped-string "']"
escaped-string   = *(unescaped-char / "''" )
unescaped-char   = %x20-26 / %x28-7E / UTF8-multibyte
                 ; any printable character except single quote (0x27), plus UTF-8

array-index      = "[" non-neg-integer "]"
non-neg-integer  = "0" / (DIGIT1 *DIGIT)

key-filter       = "[?(@" member-access "==" filter-literal ")]"
member-access    = 1*("." property-name / "['" escaped-string "']")

value-filter     = "[?(@==" filter-literal ")]"

filter-literal   = string-literal / number-literal / boolean-literal / null-literal
string-literal   = "'" escaped-string "'"
number-literal   = ["-"] ("0" / DIGIT1 *DIGIT) ["." 1*DIGIT] [("e" / "E") ["+" / "-"] 1*DIGIT]
boolean-literal  = "true" / "false"
null-literal     = "null"

ALPHA  = %x41-5A / %x61-7A
DIGIT  = %x30-39
DIGIT1 = %x31-39
```

A formal ABNF grammar is provided in Appendix C.

### 5.2 Productions

| # | Production | Syntax | Example |
| --- | --- | --- | --- |
| 1 | Root | `$` | `$` |
| 2 | Dot property | `.name` | `$.user.name` |
| 3 | Bracket property | `['key']` | `$['a.b']` |
| 4 | Array index | `[n]` | `$.items[0]` |
| 5 | Key filter | `[?(@.key==val)]` | `$.items[?(@.id==42)]`, `$.orders[?(@.item.id==1)]` |
| 6 | Value filter | `[?(@==val)]` | `$.tags[?(@=='urgent')]` |

The following examples use this document to illustrate each production:

```json
{
  "user": { "name": "Alice", "address": { "city": "Portland" } },
  "items": [
    { "id": 1, "name": "Widget", "price": 10 },
    { "id": 2, "name": "Gadget", "price": 20 }
  ],
  "orders": [
    { "item": { "id": 1 }, "qty": 3 },
    { "item": { "id": 2 }, "qty": 1 }
  ],
  "tags": ["urgent", "review", "draft"],
  "config": { "a.b": 1 }
}
```

**1. Root** (`$`) — resolves to the entire document.

**2. Dot property** (`$.user.name`) — resolves to `"Alice"`. Multiple dot segments chain: `$.user.address.city` resolves to `"Portland"`.

Invalid path: `$.user.0` — a digit after dot notation is not valid (Section 5.5). Use `$.user['0']` for a property named `"0"`, or `$.items[0]` for an array index.

**3. Bracket property** (`$['a.b']`) — resolves to the value `1` from the property named `"a.b"` (which contains a literal dot). Bracket notation is required because dot notation would incorrectly parse as two segments.

Invalid path: `$[a.b]` — bracket properties MUST use single quotes: `$['a.b']`.

**4. Array index** (`$.tags[1]`) — resolves to `"review"` (zero-based index).

Invalid path: `$.tags[-1]` — negative indices are not supported. Indices MUST be non-negative integers.

**5. Key filter** (`$.items[?(@.id==1)]`) — resolves to `{ "id": 1, "name": "Widget", "price": 10 }`. The filter matches the element whose `id` property equals the number `1`.

Member-access in filters supports chained segments for nested property access (per RFC 9535). For example, `$.orders[?(@.item.id==1)]` resolves to `{ "item": { "id": 1 }, "qty": 3 }` — the filter traverses `element.item.id`. Each segment follows the same notation rules: dot notation for identifiers, bracket notation for non-identifiers.

Invalid path: `$.items[?(@.id=='1')]` — the filter uses a string literal `'1'`, but the document's `id` values are numbers. Filter comparison uses strict JSON value equality (Section 5.4); a string does not match a number.

**6. Value filter** (`$.tags[?(@=='urgent')]`) — resolves to `"urgent"`.

Invalid path: `$.tags[?(@==urgent)]` — string filter literals MUST be single-quoted: `$.tags[?(@=='urgent')]`.

#### 5.2.1 Deep Paths After Filters

A key filter or value filter MAY be followed by additional segments to address properties within the matched element:

- `$.items[?(@.id==1)].name` — resolves to `"Widget"` (the `name` property of the matched element).
- `$.items[?(@.id==1)].price` — resolves to `10`.

This is essential for making property-level changes to keyed-array elements without replacing the entire element. When a path has segments after a filter, the keyed-array value consistency rule (Section 6.4) does NOT apply — the operation targets a property, not the element itself.

### 5.3 Constraints

A JSON‑Atom Path MUST resolve to **exactly one location** in the document. The following JSONPath features are NOT supported:

- Wildcards (`*`)
- Recursive descent (`..`)
- Array slices (`[0:3]`)
- Multi-value selectors (`[0,1,2]`)
- Multiple filter predicates
- Function expressions
- Regular expressions in filters

### 5.4 Filter Literals

Filter expressions support four literal types, matching the JSON scalar types:

| Type | Syntax | Example |
| --- | --- | --- |
| String | Single-quoted | `[?(@.name=='Alice')]` |
| Number | JSON number | `[?(@.id==42)]` |
| Boolean | `true` / `false` | `[?(@.active==true)]` |
| Null | `null` | `[?(@.status==null)]` |

**Comparison rule:** Filter comparison uses JSON value equality — both type and value MUST match. The number `42` does NOT match the string `"42"`.

**String escaping:** Single quotes within string literals are escaped by doubling: `'O''Brien'` represents the string `O'Brien`.

**Canonical form:** The literal type in the filter MUST match the JSON type of the property value. A path `[?(@.id=='42')]` targeting a document where `id` is the number `42` is not in canonical form — the correct canonical path is `[?(@.id==42)]`.

**Example — type-sensitive comparison:**

Given document `{ "items": [{ "id": 1, "name": "Widget" }] }`:

```json
{ "op": "replace", "path": "$.items[?(@.id==1)].name", "value": "Gizmo", "oldValue": "Widget" }
```

The filter uses the number literal `1` (no quotes), matching the numeric `id` in the document. Using `'1'` (string literal) would not match because string `"1"` does not equal number `1`.

### 5.5 Canonical Form

A path is in **canonical form** when it satisfies all of the following:

1. Uses dot notation for property names matching `[a-zA-Z_][a-zA-Z0-9_]*`.
2. Uses bracket-quote notation (`$['key']`) ONLY for property names that cannot use dot notation.
3. Array indices use bracket notation: `[0]`, never `.0`.
4. Filter expressions use bracket notation: `[?(@.id==42)]`.
5. Root is `$`.

Examples:

| Canonical | Non-canonical equivalent |
| --- | --- |
| `$.user.name` | `$['user']['name']` |
| `$['a.b']` | _(no dot alternative — name contains `.`)_ |
| `$.items[0]` | _(never `$.items.0`)_ |
| `$.items[?(@.id==42)]` | `$.items[?(@['id']==42)]` |
| `$.orders[?(@.item.id==1)]` | `$.orders[?(@['item']['id']==1)]` |

**Rule:** `$.property.0` (dot followed by a digit) is NOT a valid path. Numeric segments after dot notation are ambiguous between a property name `"0"` and an array index `0`. Array indices MUST use bracket notation.

**Producer rule:** Implementations that produce deltas MUST emit canonical paths.

**Consumer rule:** Implementations that consume deltas MUST accept both canonical form and bracket-for-everything form (e.g., `$['user']['name']`).

**Example — canonical vs. non-canonical:**

These two operations are semantically identical. Producers MUST emit the canonical form; consumers MUST accept both:

Canonical (MUST produce):

```json
{ "op": "replace", "path": "$.user.name", "value": "Anna", "oldValue": "Alice" }
```

Non-canonical (MUST accept):

```json
{ "op": "replace", "path": "$['user']['name']", "value": "Anna", "oldValue": "Alice" }
```

## 6. Operations

Each operation is a JSON object within the `operations` array. Every operation MUST contain:

- `op` (string, REQUIRED): The operation type. One of `"add"`, `"remove"`, `"replace"`.
- `path` (string, REQUIRED): A JSON‑Atom Path (Section 5).

Operations MAY contain additional properties. Unknown properties MUST be preserved and MUST NOT cause errors. Properties prefixed with `x_` are reserved for extensions.

### 6.1 `add`

Adds a value at a path that does not exist in the source document.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `op` | string | REQUIRED | `"add"` |
| `path` | string | REQUIRED | JSON‑Atom Path. See target rules below. |
| `value` | any | REQUIRED | The value to add. |
| `oldValue` | | MUST NOT be present | |

**Target rules by path type:**

- **Object property** (`$.obj.newProp`): Creates the property. The property MUST NOT already exist in the source.
- **Array index** (`$.arr[n]`): Inserts the value at index `n`, shifting subsequent elements to higher indices.
- **Key filter** (`$.arr[?(@.id=='4')]`): Appends the value to the array. The filter MUST NOT match any existing element. The `value` MUST be an object containing the identity property with a value matching the filter literal (see Section 6.4).
- **Value filter** (`$.arr[?(@=='new')]`): Appends the value to the array. The filter MUST NOT match any existing element (values must be unique per Section 7.3).
- **Root** (`$`): Sets the root value. The source MUST be `null` (see Section 6.5).

#### Examples

**Add an object property:**

Given source `{ "user": { "name": "Alice" } }`:

```json
{ "op": "add", "path": "$.user.role", "value": "admin" }
```

**Invalid — property already exists:**

```json
{ "op": "add", "path": "$.user.name", "value": "Bob" }
```

The property `name` already exists. Use `replace` to change an existing value.

**Add to a keyed array:**

Given source `{ "items": [{ "id": 1, "name": "Widget", "price": 10 }] }`:

```json
{ "op": "add", "path": "$.items[?(@.id==4)]", "value": { "id": 4, "name": "Thingamajig", "price": 40 } }
```

The new element is appended to the array. The `value` contains `"id": 4` matching the filter literal `@.id==4` (Section 6.4).

**Add at root:**

Given source `null`:

```json
{ "op": "add", "path": "$", "value": { "created": true } }
```

**Invalid — `oldValue` on `add`:**

```json
{ "op": "add", "path": "$.user.role", "value": "admin", "oldValue": null }
```

`add` operations MUST NOT include `oldValue`.

### 6.2 `remove`

Removes a value at a path that exists in the source document.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `op` | string | REQUIRED | `"remove"` |
| `path` | string | REQUIRED | JSON‑Atom Path. Target MUST exist in source. |
| `oldValue` | any | OPTIONAL | The removed value. Required for reversible deltas. |
| `value` | | MUST NOT be present | |

**Target rules by path type:**

- **Object property** (`$.obj.prop`): Deletes the property.
- **Array index** (`$.arr[n]`): Removes the element at index `n`, shifting subsequent elements to lower indices.
- **Key filter** (`$.arr[?(@.id=='2')]`): Removes the matched element. The filter MUST match exactly one element.
- **Value filter** (`$.arr[?(@=='old')]`): Removes the matched element. The filter MUST match exactly one element.
- **Root** (`$`): Sets the root value to `null` (see Section 6.5).

#### Examples

**Remove an object property:**

Given source `{ "settings": { "theme": "light", "legacyMode": true } }`:

```json
{ "op": "remove", "path": "$.settings.legacyMode", "oldValue": true }
```

After applying, the `settings` object contains only `{ "theme": "light" }`.

**Remove from a keyed array:**

Given source `{ "items": [{ "id": 1, "name": "Widget" }, { "id": 2, "name": "Gadget" }] }`:

```json
{ "op": "remove", "path": "$.items[?(@.id==2)]", "oldValue": { "id": 2, "name": "Gadget" } }
```

**Invalid — filter matches nothing:**

```json
{ "op": "remove", "path": "$.items[?(@.id==99)]" }
```

No element matches `id==99`. The filter MUST match exactly one element for `remove`.

### 6.3 `replace`

Replaces the value at a path with a new value.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `op` | string | REQUIRED | `"replace"` |
| `path` | string | REQUIRED | JSON‑Atom Path. Target MUST exist in source. |
| `value` | any | REQUIRED | The new value. |
| `oldValue` | any | OPTIONAL | The previous value. Required for reversible deltas. |

**Target rules by path type:**

- **Object property** (`$.obj.prop`): Replaces the property value.
- **Array index** (`$.arr[n]`): Replaces the element at index `n`.
- **Key filter** (`$.arr[?(@.id=='1')]`): Replaces the matched element. The filter MUST match exactly one element. For element-level replacement, the `value` MUST contain the identity property matching the filter literal (see Section 6.4).
- **Value filter** (`$.arr[?(@=='old')]`): Replaces the matched element.
- **Root** (`$`): Replaces the root value (see Section 6.5).

**Type changes:** A change in value type (e.g., string to number, object to array) MUST be expressed as a single `replace` operation, NOT as a `remove` followed by an `add`.

#### Examples

**Replace an object property:**

Given source `{ "user": { "name": "Alice", "email": "alice@example.com" } }`:

```json
{ "op": "replace", "path": "$.user.name", "value": "Anna", "oldValue": "Alice" }
```

**Replace a nested object property:**

Given source `{ "user": { "name": "Alice", "address": { "city": "Portland" } } }`:

```json
{ "op": "replace", "path": "$.user.address.city", "value": "Seattle", "oldValue": "Portland" }
```

**Replace a property within a keyed-array element (deep path):**

Given source `{ "items": [{ "id": 1, "name": "Widget", "price": 10 }] }`:

```json
{ "op": "replace", "path": "$.items[?(@.id==1)].name", "value": "Widget Pro", "oldValue": "Widget" }
```

The path targets the `name` property within the matched element. Because the path has a trailing segment after the filter, the keyed-array value consistency rule (Section 6.4) does NOT apply — `value` is the property value `"Widget Pro"`, not the entire element.

**Type change (single `replace`):**

Given source `{ "metadata": null }`:

```json
{ "op": "replace", "path": "$.metadata", "value": { "updated": true }, "oldValue": null }
```

Changes `metadata` from `null` to an object in a single operation.

**Invalid — type change as `remove` + `add`:**

```json
[
  { "op": "remove", "path": "$.metadata", "oldValue": null },
  { "op": "add", "path": "$.metadata", "value": { "updated": true } }
]
```

A type change MUST be expressed as a single `replace`, not as a `remove` followed by an `add`.

**Invalid — missing `value`:**

```json
{ "op": "replace", "path": "$.user.name", "oldValue": "Alice" }
```

`replace` operations MUST include `value`.

### 6.4 Keyed-Array Value Consistency

For `add` operations and element-level `replace` operations targeting a keyed array (key filter path without a trailing property segment), the `value` MUST contain the identity property with a value matching the filter literal. Comparison uses JSON value equality (type and value must match).

**Valid:**

```json
{ "op": "add", "path": "$.items[?(@.id=='4')]", "value": { "id": "4", "name": "Widget" } }
```

The filter specifies `id=='4'` (string), and `value.id` is `"4"` (string). Match.

**Invalid:**

```json
{ "op": "add", "path": "$.items[?(@.id=='4')]", "value": { "id": 4, "name": "Widget" } }
```

The filter specifies `id=='4'` (string), but `value.id` is `4` (number). Type mismatch.

For operations targeting a property within a keyed-array element (e.g., `$.items[?(@.id=='4')].name`), this constraint does not apply — the `value` is the property value, not the element.

Conformant producers MUST satisfy this rule. Conformant consumers SHOULD validate it.

### 6.5 Root Operations

Operations targeting the root path `$` affect the root JSON value itself.

At the JSON level, the source of any delta is a JSON value. The value `null` is used to represent the absence of meaningful content. Whether `null` maps to "no document" or "empty document" is an application-level concern outside the scope of this specification.

| Operation | Precondition | Result |
| --- | --- | --- |
| `add` (`$`) | Source MUST be `null`. | The root value becomes `value`. |
| `remove` (`$`) | Source MUST NOT be `null`. | The root value becomes `null`. |
| `replace` (`$`) | Source MUST NOT be `null`. | The root value becomes `value`. |

**Example — set the root value (from `null`):**

Given source `null`:

```json
{ "op": "add", "path": "$", "value": { "name": "Alice", "active": true } }
```

The result is `{ "name": "Alice", "active": true }`.

**Example — replace the root value:**

Given source `{ "name": "Alice" }`:

```json
{ "op": "replace", "path": "$", "value": { "name": "Bob", "role": "admin" }, "oldValue": { "name": "Alice" } }
```

The result is `{ "name": "Bob", "role": "admin" }`.

**Example — reset the root value to `null`:**

Given source `{ "name": "Alice" }`:

```json
{ "op": "remove", "path": "$", "oldValue": { "name": "Alice" } }
```

The result is `null`.

Note: Applications that model document lifecycle (creation, deletion) MAY map `add` at `$` to "create document" and `remove` at `$` to "delete document." This mapping is an application concern; the specification defines only the JSON-level transformation.

## 7. Array Identity Model

JSON‑Atom Format supports three models for identifying array elements:

### 7.1 Index-Based Identity

**Syntax:** `$.arr[n]` (zero-based integer index)

Operations address elements by their position. Operations apply to the document state after all prior operations in the delta have been applied. This means indices are sensitive to preceding `add` and `remove` operations that shift elements.

Index-based identity is order-sensitive and positional. It is the simplest model but the most fragile across concurrent changes.

**Example:**

```json
{ "op": "replace", "path": "$.tags[1]", "value": "approved", "oldValue": "review" }
```

### 7.2 Key-Based Identity

**Syntax:** `$.arr[?(@.key==value)]`

Operations address elements by a property value within each element. The identity key (e.g., `id`, `name`) is encoded directly in the path filter expression. No external schema is needed to interpret the delta.

Key-based identity is order-independent. The identity property MUST have unique values within the array. If a key filter matches more than one element, the operation MUST fail (Section 8.2). Comparison uses JSON value equality (strict type matching).

**Example:**

```json
{ "op": "replace", "path": "$.items[?(@.id==1)].price", "value": 15, "oldValue": 10 }
```

Addresses the item by its `id` value, independent of position in the array.

### 7.3 Value-Based Identity

**Syntax:** `$.arr[?(@==value)]`

Operations address elements of primitive arrays by their value. Used for arrays of strings, numbers, booleans, or null values.

Values MUST be unique within the array. If a value filter matches more than one element, the operation MUST fail (Section 8.2). Comparison uses JSON value equality.

**Example:**

```json
{ "op": "remove", "path": "$.tags[?(@=='draft')]", "oldValue": "draft" }
```

### 7.4 Recommendations

- Implementations SHOULD prefer key-based identity when the array elements have a suitable identity property.
- Within a single delta, all operations targeting the same array SHOULD use the same identity model.
- Key-based `add` operations append the new element to the end of the array. If element ordering matters beyond identity, use index-based operations.

## 8. Applying a Delta

Operations in a delta are applied **sequentially in array order**. Each operation modifies the document; subsequent operations see the modified state. This is identical to the application semantics defined in [RFC 6902 Section 3](https://www.rfc-editor.org/rfc/rfc6902#section-3).

### 8.1 Application Rule

```
apply(source, delta) == target
```

A conformant implementation MUST produce the correct target document when applying a valid delta to the corresponding source document.

### 8.2 Error Conditions

The following conditions MUST be treated as errors:

- Path resolves to a location that does not exist (for `replace` and `remove`)
- Path resolves to a location that already exists (for `add` on object properties)
- Key filter matches zero elements (for `replace`, `remove`)
- Key filter matches two or more elements (for any operation)
- Value filter matches zero or multiple elements (for `replace`, `remove`)
- Malformed path expression
- Keyed-array value consistency violation (Section 6.4)
- Missing required fields (`op`, `path`, `value` where applicable)
- Unknown `op` value

Implementations MUST NOT silently corrupt the document on error. Implementations SHOULD stop processing and report the error. The state of the document after an error is implementation-defined.

**Example error — filter matches zero elements:**

Given source `{ "items": [{ "id": 1, "name": "Widget" }] }`:

```json
{ "op": "remove", "path": "$.items[?(@.id==99)]" }
```

No element matches `id==99`. Implementations MUST treat this as an error.

**Example error — unknown operation type:**

```json
{ "op": "move", "path": "$.user.name", "value": "$.user.fullName" }
```

`"move"` is not a defined operation. Implementations MUST reject unknown `op` values.

### 8.3 Worked Example

Source document:

```json
{
  "user": { "name": "Alice" },
  "items": [
    { "id": 1, "name": "Widget", "price": 10 },
    { "id": 2, "name": "Gadget", "price": 20 }
  ]
}
```

Delta:

```json
{
  "format": "json-atom",
  "version": 1,
  "operations": [
    { "op": "replace", "path": "$.user.name", "value": "Anna", "oldValue": "Alice" },
    { "op": "remove", "path": "$.items[?(@.id==2)]", "oldValue": { "id": 2, "name": "Gadget", "price": 20 } },
    { "op": "add", "path": "$.items[?(@.id==3)]", "value": { "id": 3, "name": "Doohickey", "price": 30 } }
  ]
}
```

Step-by-step application:

1. **Operation 1** (`replace $.user.name`): `"Alice"` → `"Anna"`.
2. **Operation 2** (`remove $.items[?(@.id==2)]`): Element `{ "id": 2, "name": "Gadget", "price": 20 }` is removed from `items`.
3. **Operation 3** (`add $.items[?(@.id==3)]`): Element `{ "id": 3, "name": "Doohickey", "price": 30 }` is appended to `items`.

Target document:

```json
{
  "user": { "name": "Anna" },
  "items": [
    { "id": 1, "name": "Widget", "price": 10 },
    { "id": 3, "name": "Doohickey", "price": 30 }
  ]
}
```

## 9. Reversibility

### 9.1 Definition

A delta is **reversible** when every `replace` and `remove` operation includes an `oldValue` field. No envelope-level flag is needed — reversibility is a derived property of the operations.

### 9.2 Computing the Inverse

The inverse of a reversible delta is computed by:

1. Reversing the order of operations.
2. Transforming each operation:

| Original | Inverse |
| --- | --- |
| `add(path, value)` | `remove(path, oldValue=value)` |
| `remove(path, oldValue)` | `add(path, value=oldValue)` |
| `replace(path, value, oldValue)` | `replace(path, value=oldValue, oldValue=value)` |

**Example:**

Original delta:

```json
{
  "format": "json-atom",
  "version": 1,
  "operations": [
    { "op": "replace", "path": "$.user.name", "value": "Anna", "oldValue": "Alice" },
    { "op": "add", "path": "$.user.role", "value": "admin" }
  ]
}
```

Inverse delta (operations reversed, each operation transformed):

```json
{
  "format": "json-atom",
  "version": 1,
  "operations": [
    { "op": "remove", "path": "$.user.role", "oldValue": "admin" },
    { "op": "replace", "path": "$.user.name", "value": "Alice", "oldValue": "Anna" }
  ]
}
```

The `add` became a `remove` (with the added value as `oldValue`), and the `replace` swapped its `value` and `oldValue`. The order is reversed.

### 9.3 Round-Trip Guarantee

For a reversible delta, applying the inverse to the target MUST recover the source:

```
apply(target, inverse(delta)) == source
```

The equality comparison uses JSON value equality: objects are compared by key-value pairs (unordered), and arrays are compared element-by-element **in order**.

### 9.4 Round-Trip and Array Ordering

Key-based `add` operations append elements to the end of the array (Section 6.1). When a delta removes an element from the middle of a keyed array and the inverse re-adds it, the element will be appended rather than restored to its original position. In this case, the round-trip result is **value-equivalent** to the source (same elements, same identity keys) but MAY differ in array element order.

The strict round-trip equation in Section 9.3 holds with positional array equality when:

- The delta uses only object property operations (no array ordering concern).
- The delta uses index-based array operations (position is explicit).
- Key-based array `remove` operations target elements at the end of the array.

For keyed arrays where element order is not semantically significant, implementations MAY treat the round-trip as successful when the arrays contain the same elements regardless of order.

Implementations that require position-preserving round-trips for arrays SHOULD use index-based identity.

### 9.5 Validation

Consumers that require reversibility SHOULD validate `oldValue` presence on all `replace` and `remove` operations at ingest time.

## 10. Conformance

### 10.1 Level 1: Apply

A Level 1 conformant implementation:

- Can parse a delta document (Section 4)
- Can parse JSON‑Atom Path expressions (Section 5)
- Can apply a delta to a source document and produce the correct target document (Section 8)
- MUST pass all Level 1 conformance fixtures

### 10.2 Level 2: Reversible

A Level 2 conformant implementation satisfies all Level 1 requirements and additionally:

- Can compute the inverse of a reversible delta (Section 9)
- Can apply the inverse to the target document to recover the source document
- MUST pass all Level 2 conformance fixtures

### 10.3 Conformance Testing

Conformance is verified against the fixture suite in the `fixtures/` directory. Each fixture specifies a source document, a target document, a delta, and a conformance level.

For Level 1: `apply(fixture.source, fixture.delta) == fixture.target`

For Level 2: `apply(fixture.target, inverse(fixture.delta)) == fixture.source`

### 10.4 Compute Implementations

Implementations that compute deltas (diff algorithms) are not directly covered by conformance levels, as different algorithms may produce different valid deltas for the same source/target pair.

Non-normative recommendation: compute implementations SHOULD satisfy `apply(source, compute(source, target)) == target` for all inputs.

## 11. Extensibility

### 11.1 Property Preservation

Unknown properties MUST be preserved by conformant implementations and MUST NOT cause errors.

### 11.2 Extension Placement

Extension properties MAY appear at two levels:

1. **On the delta envelope** — metadata that applies to the entire delta (e.g., author, timestamp, workflow context).
2. **On individual operations** — metadata that applies to a specific operation (e.g., comparison strategy, change reason).

Both levels follow the same preservation and namespace rules.

**Envelope-level extension:**

```json
{
  "format": "json-atom",
  "version": 1,
  "operations": [],
  "x_author": "jane@example.com",
  "x_timestamp": "2025-01-15T10:30:00Z"
}
```

**Operation-level extension:**

```json
{
  "op": "replace",
  "path": "$.summary",
  "value": "New text",
  "oldValue": "Old text",
  "x_comparison": { "strategy": "semantic-embedding", "similarity": 0.42 }
}
```

### 11.3 Extension Namespace

Properties prefixed with `x_` are reserved for extensions. Future versions of this specification will NEVER define a property starting with `x_`. This is the guaranteed-safe namespace for extension authors.

Properties without the `x_` prefix that are not defined by the current specification version are also preserved, but MAY be claimed by future specification versions. Extension authors who want guaranteed future-safety MUST use the `x_` prefix.

### 11.4 Semantic Constraints

Extensions MUST NOT alter the semantics of core properties (`op`, `path`, `value`, `oldValue`). A conformant Level 1 implementation that ignores all extension properties MUST still correctly apply the delta. Extension properties are always advisory — they provide context, not instructions.

### 11.5 Design Guidance

The delta format describes **what changed**. Extensions MAY describe **why** or **how** a change was detected, but this metadata never affects core application semantics.

See Appendix F for non-normative extension patterns covering comparison metadata, audit trails, AI agent workflows, content-addressable verification, and logical change grouping.

## 12. Security Considerations

### 12.1 Untrusted Input

Deltas from untrusted sources SHOULD be validated before application:

- Validate that all paths are well-formed and do not escape the document boundary.
- Limit maximum nesting depth to prevent stack overflow.
- Limit maximum payload size to prevent memory exhaustion.
- Validate `op` values against the defined set.

### 12.2 Path Traversal

Implementations MUST NOT allow paths to reference locations outside the document boundary. JSON‑Atom Path does not support parent references or absolute filesystem paths.

### 12.3 No Code Execution

Delta documents are pure data. Implementations MUST NOT execute any content within a delta as code. Path expressions, values, and extension properties are always data, never code.

## Appendix B: Comparison with RFC 6902 and RFC 7396

### B.1 RFC 6902 (JSON Patch)

| Aspect | RFC 6902 | JSON‑Atom Format |
| --- | --- | --- |
| Path syntax | JSON Pointer (RFC 6901) | JSON‑Atom Path (JSONPath subset) |
| Array addressing | Index only (`/items/0`) | Index, key filter, value filter |
| Operations | add, remove, replace, move, copy, test | add, remove, replace |
| Reversibility | Not supported | Supported via `oldValue` |
| Self-identifying | No | Yes (`format` field) |
| Extensibility | No | Yes (`x_` properties) |
| Type change | replace | replace |

JSON‑Atom Format intentionally omits `move`, `copy`, and `test` operations. `move` and `copy` can be expressed as `remove` + `add` or `add` with the same value. `test` is an assertion, not a change, and is out of scope for a change format.

### B.2 RFC 7396 (JSON Merge Patch)

| Aspect | RFC 7396 | JSON‑Atom Format |
| --- | --- | --- |
| Format | Partial JSON document | Operation array |
| Atomic operations | No | Yes |
| Array changes | Whole-array replacement only | Element-level operations |
| `null` handling | `null` means delete | `null` is a valid value |
| Reversibility | Not possible | Supported via `oldValue` |
| Nested changes | Implicit via structure | Explicit via paths |

## Appendix C: JSON‑Atom Path ABNF Grammar

```abnf
; JSON‑Atom Path — Formal Grammar
; Conforms to RFC 5234 (ABNF)

json-atom-path = root *segment

root = "$"

segment = dot-property
        / bracket-property
        / array-index
        / key-filter
        / value-filter

; Dot property: $.name
dot-property = "." property-name
property-name = (ALPHA / "_") *(ALPHA / DIGIT / "_")

; Bracket property: $['key']
bracket-property = "['" escaped-string "']"

; Array index: $[0]
array-index = "[" non-neg-integer "]"
non-neg-integer = "0" / (DIGIT1 *DIGIT)
DIGIT1 = %x31-39  ; 1-9

; Key filter: $[?(@.key==value)], $[?(@.a.b==value)], or $[?(@['dotted.key']==value)]
key-filter = "[?(@" member-access "==" filter-literal ")]"
member-access = 1*("." property-name / "['" escaped-string "']")

; Value filter: $[?(@==value)]
value-filter = "[?(@==" filter-literal ")]"

; Filter literals
filter-literal = string-literal
               / number-literal
               / boolean-literal
               / null-literal

string-literal = "'" escaped-string "'"
escaped-string = *(unescaped-char / escaped-quote)
unescaped-char = %x20-26 / %x28-7E / UTF8-2 / UTF8-3 / UTF8-4
               ; printable ASCII except single quote (0x27), plus UTF-8
escaped-quote  = "''"  ; two single quotes represent one single quote

number-literal = [ "-" ] int [ frac ] [ exp ]
int            = "0" / (DIGIT1 *DIGIT)
frac           = "." 1*DIGIT
exp            = ("e" / "E") [ "+" / "-" ] 1*DIGIT

boolean-literal = "true" / "false"
null-literal    = "null"

; UTF-8 multibyte sequences (RFC 3629)
UTF8-2 = %xC2-DF UTF8-tail
UTF8-3 = %xE0 %xA0-BF UTF8-tail
       / %xE1-EC 2UTF8-tail
       / %xED %x80-9F UTF8-tail
       / %xEE-EF 2UTF8-tail
UTF8-4 = %xF0 %x90-BF 2UTF8-tail
       / %xF1-F3 3UTF8-tail
       / %xF4 %x80-8F 2UTF8-tail
UTF8-tail = %x80-BF
```

## Appendix D: Reference Path Parser (Python-Style Pseudocode)

The following pseudocode demonstrates a minimal JSON‑Atom Path parser. The grammar is simple enough that a full parser can be implemented in approximately 30 lines in most languages.

```python
def parse_json_delta_path(path: str) -> list[Segment]:
    """Parse a JSON‑Atom Path into a list of segments."""
    segments = []
    i = 0

    if not path.startswith('$'):
        raise ParseError("Path must start with '$'")
    i = 1  # skip '$'

    while i < len(path):
        if path[i] == '.':
            # Dot property: .name
            i += 1
            start = i
            while i < len(path) and (path[i].isalnum() or path[i] == '_'):
                i += 1
            segments.append(DotProperty(path[start:i]))

        elif path[i] == '[':
            if path[i+1] == '?':
                # Filter: [?(@.key==val)] or [?(@['key']==val)] or [?(@==val)]
                # Find the closing )] — safe because ')' doesn't appear in literals
                end = path.index(')]', i)
                inner = path[i+3:end]  # strip "[?(" and will strip ")" later
                segments.append(parse_filter(inner))
                i = end + 2

            elif path[i+1] == "'":
                # Bracket property: ['key']
                key, end = extract_quoted_string(path, i+2)  # after ['
                segments.append(BracketProperty(key))
                i = end + 2  # skip ']

            elif path[i+1].isdigit():
                # Array index: [0]
                end = path.index(']', i)
                index = int(path[i+1:end])
                segments.append(ArrayIndex(index))
                i = end + 1

            else:
                raise ParseError(f"Unexpected character after '[': {path[i+1]}")
        else:
            raise ParseError(f"Unexpected character: {path[i]}")

    return segments


def parse_filter(inner: str) -> Filter:
    """Parse filter content: @.id==42, @['a.b']==42, or @=='value'."""
    if inner.startswith('@.'):
        # Key filter with dot property: @.key==val
        eq = inner.index('==')
        key = inner[2:eq]
        return KeyFilter(key, parse_literal(inner[eq+2:]))
    elif inner.startswith("@['"):
        # Key filter with bracket property: @['dotted.key']==val
        key, end = extract_quoted_string(inner, 3)  # after @['
        # inner[end] should be ']', then '=='
        val_start = end + 3  # skip ']==
        return KeyFilter(key, parse_literal(inner[val_start:]))
    elif inner.startswith('@=='):
        # Value filter: @==val
        return ValueFilter(parse_literal(inner[3:]))
    else:
        raise ParseError(f"Invalid filter: {inner}")


def parse_literal(s: str):
    """Parse a filter literal: 'string', number, true, false, null."""
    if s.startswith("'") and s.endswith("'"):
        return s[1:-1].replace("''", "'")  # unescape doubled quotes
    elif s == 'true':
        return True
    elif s == 'false':
        return False
    elif s == 'null':
        return None
    else:
        return float(s) if '.' in s or 'e' in s or 'E' in s else int(s)


def extract_quoted_string(s: str, start: int) -> tuple[str, int]:
    """Extract a single-quoted string starting at index `start` (after the opening quote).
    Returns (unescaped string, index of closing quote)."""
    result = []
    i = start
    while i < len(s):
        if s[i] == "'":
            if i + 1 < len(s) and s[i+1] == "'":
                result.append("'")  # escaped quote
                i += 2
            else:
                return (''.join(result), i)  # closing quote
        else:
            result.append(s[i])
            i += 1
    raise ParseError("Unterminated quoted string")
```

**Key implementation insight:** The grammar has no ambiguity at any decision point. After `[`, the second character fully disambiguates: `?` means filter, `'` means bracket property, and a digit means array index.

## Appendix E: Migration Guide from json-diff-ts

[json-diff-ts](https://github.com/ltwlf/json-diff-ts) is the originating implementation from which the JSON‑Atom Format specification was derived. It is evolving toward full conformance. This appendix documents the mapping between the json-diff-ts internal format and the JSON‑Atom Format specification.

### E.1 Operation Mapping

| json-diff-ts | JSON‑Atom Format | Notes |
| --- | --- | --- |
| `type: 'ADD'` | `op: 'add'` | Rename + lowercase |
| `type: 'UPDATE'` | `op: 'replace'` | Rename: "update" becomes "replace" |
| `type: 'REMOVE'` | `op: 'remove'` | Rename + lowercase |

### E.2 Field Mapping (IAtomicChange to Operation)

| IAtomicChange | Operation | Transformation |
| --- | --- | --- |
| `type` | `op` | See operation mapping above |
| `key` | _(dropped)_ | Derivable from the last segment of `path` |
| `path` | `path` | `$root` sentinel rewritten to `$` |
| `valueType` | _(dropped)_ | JSON values are self-typing |
| `value` (on ADD/UPDATE) | `value` | Direct copy |
| `value` (on REMOVE) | `oldValue` | **Semantic rename** — the removed value moves from `value` to `oldValue` |
| `oldValue` (on UPDATE) | `oldValue` | Direct copy |

### E.3 Structural Changes

- **Envelope added:** JSON‑Atom Format wraps operations in `{ "format": "json-atom", "version": 1, "operations": [...] }`.
- **Type changes:** json-diff-ts may emit REMOVE + ADD pairs for type changes (when `treatTypeChangeAsReplace: false`). JSON‑Atom Format uses a single `replace` operation.
- **Hierarchical format:** json-diff-ts's `IChange[]` with nested `changes` arrays must be atomized (via `atomizeChangeset()`) before converting to JSON‑Atom Format.

### E.4 Non-JSON Values

json-diff-ts can diff JavaScript objects containing `Date`, `undefined`, `RegExp`, and other non-JSON types. JSON‑Atom Format is strict RFC 8259 JSON. When converting:

- `Date` values SHOULD be serialized to ISO 8601 strings.
- `undefined` values SHOULD be omitted (they represent absence in JSON).
- `RegExp`, `Function`, and other non-serializable types SHOULD cause an error.

### F.5 Logical Change Grouping

In real applications, atomized operations often originate from a single higher-level change action — a user edit, a workflow step, or an API call. The `x_changeId` extension groups related operations into a logical unit.

**Envelope-level grouping** — all operations in the delta belong to the same logical change:

```json
{
  "format": "json-atom",
  "version": 1,
  "x_changeId": "chg_abc123",
  "x_source": "user-profile-form",
  "operations": [
    { "op": "replace", "path": "$.user.name", "value": "Anna", "oldValue": "Alice" },
    { "op": "add", "path": "$.user.role", "value": "admin" }
  ]
}
```

**Operation-level grouping** — individual operations carry grouping metadata, useful when operations from different logical changes are stored together:

```json
{
  "op": "replace",
  "path": "$.user.name",
  "value": "Anna",
  "oldValue": "Alice",
  "x_changeId": "chg_abc123"
}
```

Logical change grouping does not affect delta semantics. It is useful for:

- **Audit logs** — correlating operations to the user action or workflow step that produced them.
- **Replay systems** — replaying or reverting a logical change as a unit.
- **Atomized storage** — reconstructing complete delta documents from individually stored operations (see F.6).

### F.6 Storage and Atomization

The JSON‑Atom Format specification defines the delta document format. It does not prescribe how deltas are stored or transmitted. Implementations may choose different storage strategies depending on their needs:

- **Document storage** — store complete delta documents as defined in Section 4. Simple and self-contained.
- **Atomized storage** — store individual operations as separate records. Enables fine-grained querying, indexing, and selective replay.
- **Hybrid** — store complete deltas but index individual operations for query purposes.

When using atomized storage, grouping metadata such as `x_changeId` (Section F.5) enables reconstruction of complete delta documents from individually stored operations. An implementation might query all operations with a given `x_changeId`, order them by sequence number, and wrap them in a delta envelope to produce a valid delta document.

## Appendix G: Implementation Guidance (Non-normative)

This appendix provides practical guidance for implementers. It is non-normative; the normative rules are in the body of the specification.

### G.1 Path Parsing Edge Cases

See Appendix D for a complete reference path parser in Python-style pseudocode. The grammar has no ambiguity at any decision point — after `[`, the next character fully disambiguates the segment type.

The following edge cases deserve attention:

| Input | Result | Reason |
| --- | --- | --- |
| `$.` (trailing dot) | Parse error | Empty property name after dot |
| `$[01]` | Parse error | Leading zeros are not valid; use `$[1]` |
| `$['O''Brien']` | Property `O'Brien` | Doubled single quotes are the escape sequence |
| `$['0']` | Property named `"0"` | Bracket-quoted — this is an object property |
| `$[0]` | Array index 0 | Unquoted digit — this is an array index |
| `$.items.0` | Parse error | Digit after dot is ambiguous (Section 5.5); use `$.items[0]` |

### G.2 Filter Resolution

When resolving a key filter like `$.items[?(@.id==1)]`:

1. Navigate to `$.items` to get the target array.
2. Iterate over elements. For each element, read the `id` property and compare against the literal `1` using JSON value equality.
3. Assert the match count:
   - **`remove` / `replace`:** Exactly one match. Zero or multiple matches is an error.
   - **`add`:** Zero matches (the element must not already exist). One or more matches is an error.
4. If additional segments follow the filter (e.g., `.name`), continue navigating into the matched element.

**Comparison semantics — strict type matching:**

| Filter literal | Document value | Match? | Reason |
| --- | --- | --- | --- |
| `42` (number) | `42` (number) | Yes | Same type and value |
| `'42'` (string) | `42` (number) | No | String != Number |
| `42` (number) | `"42"` (string) | No | Number != String |
| `true` (boolean) | `"true"` (string) | No | Boolean != String |
| `null` (null) | _(property absent)_ | No | `null` is a value; absent is not `null` |

Implementations MUST NOT perform type coercion during filter comparison.

### G.3 Error Handling

The recommended validation strategy:

1. **Validate the delta structure:** Check that `format` is `"json-atom"`, `version` is a supported version, and `operations` is an array.
2. **Validate each operation:** Check that `op` is a known value, `path` is well-formed, and required fields (`value` for `add`/`replace`) are present.
3. **Apply operations sequentially.** If any operation fails, stop and report the error.

Whether to roll back partially-applied operations or leave the document in a partially-modified state is implementation-defined. Implementations that need atomicity SHOULD work on a copy of the document.

**Example error report structure (non-normative):**

```json
{
  "error": "FILTER_NO_MATCH",
  "operationIndex": 2,
  "path": "$.items[?(@.id==99)]",
  "op": "remove",
  "message": "Key filter matched zero elements"
}
```

### G.4 Common Implementation Pitfalls

#### G.4.1 Multiple Filter Matches

A key filter or value filter MUST match exactly one element (for `remove` and `replace`) or zero elements (for `add`). A common bug is to silently apply the operation to the first match when multiple elements share the same key value.

Given a document with duplicate `id` values (which itself violates the uniqueness constraint of Section 7.2):

```json
{ "items": [{ "id": 1, "name": "A" }, { "id": 1, "name": "B" }] }
```

The path `$.items[?(@.id==1)]` matches two elements. Implementations MUST report this as an error, not silently pick one.

#### G.4.2 Type Mismatches in Filter Literals

Filter comparison uses JSON value equality with strict type matching. A common mistake is coercing types during comparison:

| Filter | Document value | Match? |
| --- | --- | --- |
| `[?(@.id==42)]` | `"id": 42` | Yes |
| `[?(@.id=='42')]` | `"id": 42` | **No** — string `"42"` != number `42` |
| `[?(@.id==42)]` | `"id": "42"` | **No** — number `42` != string `"42"` |
| `[?(@.active==true)]` | `"active": "true"` | **No** — boolean != string |
| `[?(@.active==true)]` | `"active": 1` | **No** — boolean != number |

Implementations MUST NOT perform type coercion during filter comparison.

#### G.4.3 Keyed-Array Identity Mismatch

When an `add` or element-level `replace` targets a keyed array, the `value` object MUST contain the identity property with a matching value (Section 6.4). Common mistakes:

**Missing identity property:**

```json
{ "op": "add", "path": "$.items[?(@.id==5)]", "value": { "name": "Orphan" } }
```

The `value` does not contain an `id` property. This is invalid.

**Wrong identity value:**

```json
{ "op": "add", "path": "$.items[?(@.id==5)]", "value": { "id": 99, "name": "Wrong" } }
```

The filter specifies `id==5` but `value.id` is `99`. This is invalid.

**Type mismatch in identity value:**

```json
{ "op": "add", "path": "$.items[?(@.id==5)]", "value": { "id": "5", "name": "Wrong type" } }
```

The filter specifies `id==5` (number) but `value.id` is `"5"` (string). This is invalid — comparison uses strict JSON value equality.

#### G.4.4 Null vs. Missing Properties

JSON‑Atom Format distinguishes between `null` (a JSON value) and absent (a property that does not exist in the object). These are different states that require different operations:

**Setting a property to `null`:**

```json
{ "op": "replace", "path": "$.user.email", "value": null, "oldValue": "alice@example.com" }
```

After applying, the document contains `"email": null`.

**Removing a property entirely:**

```json
{ "op": "remove", "path": "$.user.email", "oldValue": "alice@example.com" }
```

After applying, the document does NOT contain the `email` property at all.

Implementations in languages where `null` and "absent" are conflated (e.g., JavaScript's `undefined` vs. `null`, or languages where accessing a missing key returns `null`) must take care to distinguish these cases.
